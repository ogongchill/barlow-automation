구현 요청을 분석하여 실행 가능한 계획을 수립한다.
Orchestrator가 호출하며, 출력은 Codegen이 그대로 사용할 수 있는 구조화된 계획이어야 한다.

## 입력

`$ARGUMENTS` — 구현 요청 내용 (자연어 또는 기능 명세)

## 작업 절차

### 1. 컨텍스트 수집
다음 파일을 반드시 읽는다:
- `CLAUDE.md` — 공통 원칙 확인
- `readme.md` — 전체 요구사항 확인
- 요청과 관련된 `src/` 하위 기존 코드 (변경 대상 파일)

### 2. 모듈 매핑
요청이 어느 모듈에 해당하는지 판단하고 대응하는 skill을 선택한다:

| 모듈 | 해당 skill |
|------|-----------|
| Slack 이벤트, 핸들러, 라우터 | `implement-slack` |
| 세션 관리, 동시성 제어 | `implement-session` |
| Agent, LLM runner, 역할별 Agent | `implement-agent` |
| 로그, trace | `implement-logging` |
| 복합 (두 모듈 이상) | 해당 skill 목록 모두 명시 |

### 3. 계획 수립 기준
- 변경/생성이 필요한 파일을 구체적으로 명시
- 파일 간 의존 순서를 고려한 작업 순서 결정
- 인터페이스 변경이 필요한 경우 별도 task로 분리
- 기존 코드를 재사용할 수 있는 부분 명시

## 출력 형식

반드시 아래 형식으로 출력한다. 이 형식이 Codegen의 입력이 된다.

```
## 구현 계획

### 요청 요약
{요청 내용을 한 문장으로 요약}

### 적용 Skill
- {skill 이름} — {이유}

### 영향 범위
- 생성: {파일 경로}
- 수정: {파일 경로}
- 유지: {관련 파일 (변경 없음)}

### 작업 목록
1. {파일}: {구체적인 구현 내용}
2. {파일}: {구체적인 구현 내용}
...

### 인터페이스 계약
{모듈 간 의존성 및 주입 방식 — 변경이 있을 경우만 명시}

### 검증 기준
- {구현 완료 여부를 확인할 수 있는 조건}
```

## 제약
- 계획에 구현 코드를 포함하지 않는다
- 요청 범위를 벗어난 추가 개선을 계획에 포함하지 않는다
- CLAUDE.md의 공통 원칙을 위반하는 구조를 계획하지 않는다
